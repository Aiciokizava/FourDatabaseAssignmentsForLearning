## Описание проекта

Проект представляет собой решение аналитических SQL-задач на четырёх базах данных MySQL. Задачи охватывают широкий спектр SQL-конструкций: от базовых выборок с фильтрацией до рекурсивных обобщённых табличных выражений (CTE). Каждое решение оформлено в виде одного SQL-запроса.

## 1. База данных Vehicles (Транспортные средства)

### Схема

Четыре таблицы: `Vehicle` (производитель, модель, тип), `Car` (VIN, объём двигателя, мощность, цена, трансмиссия), `Motorcycle` (VIN, объём двигателя, мощность, цена, тип мотоцикла), `Bicycle` (серийный номер, количество передач, цена, тип велосипеда). Таблицы `Car`, `Motorcycle` и `Bicycle` связаны с `Vehicle` через внешний ключ `model`.

---

### Задача 1

**Условие:** Найти производителей и модели мотоциклов с мощностью > 150 л.с., ценой < 20 000$ и типом Sport. Отсортировать по мощности убывающе.

**Как работает запрос:**

Запрос выполняет соединение таблицы `Motorcycle` с таблицей `Vehicle` по полю `model`, чтобы получить имя производителя (`maker`). В блоке `WHERE` применяются три фильтра одновременно:

- `m.horsepower > 150` — отсекает маломощные мотоциклы
- `m.price < 20000` — ограничивает по цене
- `m.type = 'Sport'` — оставляет только спортивные

Все три условия объединены через `AND`, поэтому строка попадает в результат только при выполнении всех трёх. Сортировка `ORDER BY m.horsepower DESC` выводит самые мощные модели первыми.

Из тестовых данных только Yamaha YZF-R1 (200 л.с., 17 000$, Sport) удовлетворяет всем условиям. Kawasaki Ninja не проходит — мощность ровно 150, а условие строгое (> 150).

---

### Задача 2

**Условие:** Собрать в единый набор данных автомобили (мощность > 150, объём < 3 л, цена < 35 000$), мотоциклы (мощность > 150, объём < 1.5 л, цена < 20 000$) и велосипеды (передач > 18, цена < 4 000$). Для велосипедов мощность и объём — NULL. Сортировка по мощности убывающе.

**Как работает запрос:**

Запрос состоит из трёх блоков `SELECT`, объединённых через `UNION ALL`:

1. **Блок автомобилей** — соединяет `Car` с `Vehicle`, фильтрует по трём критериям, добавляет литеральный столбец `'Car' AS vehicle_type`.
2. **Блок мотоциклов** — аналогичная структура с другими пороговыми значениями и типом `'Motorcycle'`.
3. **Блок велосипедов** — соединяет `Bicycle` с `Vehicle`, но вместо `horsepower` и `engine_capacity` подставляет `NULL`, поскольку у велосипедов этих характеристик нет. Тип — `'Bicycle'`.

Используется именно `UNION ALL`, а не `UNION`, потому что дубликатов между разными типами транспорта быть не может, а `UNION ALL` работает быстрее (не выполняет дедупликацию).

Финальная сортировка `ORDER BY horsepower DESC` применяется ко всему объединённому результату. Строки с `NULL` (велосипеды) оказываются внизу, так как MySQL при сортировке по убыванию помещает NULL-значения в конец.

## 2. База данных Car Racing (Автогонки)

### Схема

Четыре таблицы: `Classes` (класс, тип, страна, кол-во дверей, объём двигателя, вес), `Cars` (имя, класс, год), `Races` (название, дата), `Results` (автомобиль, гонка, позиция). Связи: `Cars.class → Classes.class`, `Results.car → Cars.name`, `Results.race → Races.name`.

---

### Задача 1

**Условие:** Для каждого класса найти автомобиль с наименьшей средней позицией в гонках.

**Как работает запрос:**

Запрос использует двухуровневую структуру подзапросов:

1. **Внутренний подзапрос `ca`** — соединяет `Cars` с `Results` и группирует по `name` и `class`. Для каждого автомобиля вычисляет `AVG(r.position)` (средняя позиция) и `COUNT(r.race)` (количество гонок).

2. **Коррелированный подзапрос в WHERE** — для каждой строки из `ca` находит минимальную среднюю позицию среди всех автомобилей того же класса (`WHERE class_avgs.class = ca.car_class`). Условие `ca.average_position = MIN(avg_pos)` оставляет только лидеров класса.

Коррелированный подзапрос выполняется для каждой строки внешнего запроса, сравнивая среднюю позицию текущего автомобиля с лучшей в его классе. Поскольку в тестовых данных каждый класс содержит по одному автомобилю, все 8 автомобилей попадают в результат.

---

### Задача 2

**Условие:** Найти один автомобиль с наименьшей средней позицией среди всех. При равенстве — выбрать по алфавиту.

**Как работает запрос:**

Запрос соединяет три таблицы: `Cars → Results → Classes`. Группировка по `name`, `class`, `country` позволяет вычислить агрегаты `AVG(r.position)` и `COUNT(r.race)` для каждого автомобиля. Сортировка `ORDER BY average_position ASC, c.name ASC` ставит лучшую позицию первой, а при равенстве — алфавитный порядок. `LIMIT 1` отсекает всё, кроме первой строки.

Ferrari 488 и Ford Mustang оба имеют среднюю позицию 1.0, но Ferrari идёт раньше по алфавиту (F < F, но "Fe" < "Fo"), поэтому именно он попадает в результат.

---

### Задача 3

**Условие:** Найти классы с наименьшей средней позицией и вывести все автомобили из этих классов.

**Как работает запрос:**

Запрос использует три уровня вложенности:

1. **Самый глубокий подзапрос `min_avg`** — вычисляет `AVG(position)` для каждого класса, затем внешний `SELECT MIN(avg_pos2)` находит минимум среди всех классов. Результат — одно число (1.0).

2. **Средний подзапрос `class_avg`** — вычисляет среднюю позицию по классам и фильтрует: оставляет только те классы, чья средняя равна найденному минимуму.

3. **Внешний запрос** — через `HAVING c.class IN (...)` оставляет только автомобили из лучших классов. Дополнительно коррелированный подзапрос в `SELECT` считает `total_races` — общее количество гонок для класса.

В результат попадают Convertible (Ferrari 488) и SportsCar (Ford Mustang) — оба класса имеют среднюю позицию 1.0.

---

### Задача 4

**Условие:** Найти автомобили, чья средняя позиция лучше (меньше) средней по их классу. В классе должно быть минимум 2 автомобиля.

**Как работает запрос:**

В блоке `HAVING` используются два коррелированных подзапроса:

1. **Первый подзапрос** — вычисляет среднюю позицию всех автомобилей того же класса (`WHERE c2.class = c.class`). Условие `AVG(r.position) < этот_результат` оставляет только автомобили, выступающие лучше среднего.

2. **Второй подзапрос** — считает количество уникальных автомобилей в классе (`COUNT(DISTINCT c3.name)`). Условие `>= 2` исключает классы с одним автомобилем, где понятие «лучше среднего» не имеет смысла.

В тестовых данных только два класса имеют по 2 автомобиля: Sedan (BMW 3 Series — позиция 3, Audi A4 — позиция 8, среднее 5.5) и SUV (Toyota RAV4 — позиция 2, Nissan Rogue — позиция 3, среднее 2.5). BMW (3 < 5.5) и Toyota (2 < 2.5) проходят фильтр.

---

### Задача 5

**Условие:** Найти классы с наибольшим количеством автомобилей со средней позицией > 3.0 и вывести информацию об этих автомобилях.

**Как работает запрос:**

Запрос использует четыре CTE (обобщённых табличных выражения):

1. **`car_avg`** — базовая агрегация: средняя позиция и количество гонок для каждого автомобиля.

2. **`low_pos_count`** — подсчитывает количество автомобилей с `average_position >= 3.0` в каждом классе. Это число `low_position_count` используется для сортировки.

3. **`low_pos_cars`** — фильтрует автомобили со строгим условием `average_position > 3.0` (именно эти автомобили попадут в итоговый вывод).

4. **`total_races_per_class`** — считает общее количество гонок для каждого класса.

Финальный `SELECT` соединяет все CTE и таблицу `Classes` (для получения страны). Сортировка `ORDER BY low_position_count DESC, car_class ASC, average_position DESC` выводит сначала классы с наибольшим числом аутсайдеров.

Класс Sedan лидирует с `low_position_count = 2` (Audi A4 с позицией 8 попадает в вывод; BMW 3 Series с позицией 3.0 учитывается в подсчёте `>= 3.0`, но не выводится, так как `low_pos_cars` фильтрует строго `> 3.0`).

## 3. База данных Hotel Reservations (Бронирование отелей)

### Схема

Четыре таблицы: `Hotel` (ID, название, расположение), `Room` (ID, отель, тип номера, цена, вместимость), `Customer` (ID, имя, email, телефон), `Booking` (ID, номер, клиент, дата заезда, дата выезда). Связи: `Room.ID_hotel → Hotel.ID_hotel`, `Booking.ID_room → Room.ID_room`, `Booking.ID_customer → Customer.ID_customer`.

---

### Задача 1

**Условие:** Найти клиентов с более чем 2 бронированиями в более чем 1 отеле. Вывести список отелей и среднюю длительность пребывания.

**Как работает запрос:**

Запрос строит цепочку из четырёх соединений: `Customer → Booking → Room → Hotel`. Это позволяет для каждого бронирования получить информацию о клиенте и отеле.

Группировка по `c.ID_customer, c.name, c.email, c.phone` агрегирует все бронирования клиента. Вычисляются:

- `COUNT(b.ID_booking)` — общее число бронирований
- `GROUP_CONCAT(DISTINCT h.name ORDER BY h.name SEPARATOR ', ')` — список уникальных отелей через запятую. `DISTINCT` исключает повторения, `ORDER BY` обеспечивает алфавитный порядок
- `AVG(DATEDIFF(b.check_out_date, b.check_in_date))` — средняя длительность пребывания. Функция `DATEDIFF` вычисляет разницу в днях между датой выезда и заезда

Блок `HAVING` применяет два условия к агрегированным данным (в отличие от `WHERE`, который фильтрует до группировки):
- `COUNT(b.ID_booking) > 2` — более двух бронирований
- `COUNT(DISTINCT h.ID_hotel) > 1` — в более чем одном отеле

Из 10 клиентов только Bob Brown и Ethan Hunt удовлетворяют обоим условиям (по 3 бронирования в 2 разных отелях).

---

### Задача 2

**Условие:** Найти клиентов, которые одновременно: сделали > 2 бронирований в > 1 отеле И потратили > 500$.

**Как работает запрос:**

Запрос реализует пересечение двух выборок через `INNER JOIN` подзапросов:

1. **Подзапрос `q1`** — повторяет логику задачи 1: находит клиентов с > 2 бронированиями в > 1 отеле. Дополнительно вычисляет `SUM(r.price)` — общую сумму расходов.

2. **Подзапрос `q2`** — независимо находит клиентов с `SUM(r.price) > 500`.

3. **`INNER JOIN` по `ID_customer`** — оставляет только тех клиентов, которые присутствуют в обоих подзапросах. Это эквивалент логического «И» между двумя наборами условий.

Такая архитектура разделяет сложную логику на два независимых фильтра и объединяет их результаты. Bob Brown (820$) и Ethan Hunt (850$) проходят оба фильтра.

---

### Задача 3

**Условие:** Категоризировать отели по средней цене номера (Дешёвый < 175$, Средний 175–300$, Дорогой > 300$). Определить предпочитаемый тип для каждого клиента по приоритету: Дорогой > Средний > Дешёвый.

**Как работает запрос:**

Запрос использует два CTE:

1. **`hotel_category`** — для каждого отеля вычисляет `AVG(r.price)` по всем его номерам и через `CASE WHEN` присваивает категорию. Например, Grand Hotel: (150 + 200 + 350) / 3 ≈ 233 → «Средний»; Mountain Retreat: (250 + 400) / 2 = 325 → «Дорогой».

2. **`customer_hotels`** — соединяет клиентов с их бронированиями, номерами, отелями и категориями. `DISTINCT` исключает дублирование, если клиент бронировал один отель несколько раз.

3. **Финальный SELECT** — группирует по клиенту и определяет предпочтение через вложенные `CASE WHEN`:
   - `MAX(ch.category = 'Дорогой') = 1` — проверяет, есть ли хотя бы один дорогой отель. В MySQL логическое выражение возвращает 1 (true) или 0 (false), а `MAX` по группе даёт 1, если хотя бы одна строка истинна.
   - Аналогично для «Средний» — проверяется только если нет дорогих.
   - `GROUP_CONCAT(DISTINCT ch.hotel_name ORDER BY ch.hotel_name)` — список посещённых отелей.

Сортировка через `FIELD(...)` задаёт пользовательский порядок: сначала «Дешёвый», затем «Средний», затем «Дорогой». Функция `FIELD` возвращает позицию значения в списке аргументов (1, 2 или 3).

## 4. База данных Organization Structure (Организационная структура)

### Схема

Пять таблиц: `Departments` (ID, название), `Roles` (ID, название), `Employees` (ID, имя, должность, ID менеджера, ID отдела, ID роли), `Projects` (ID, название, даты, ID отдела), `Tasks` (ID, название, назначен на сотрудника, ID проекта). Ключевая особенность — самоссылающийся внешний ключ `Employees.ManagerID → Employees.EmployeeID`, формирующий иерархию подчинения.

---

### Задача 1

**Условие:** Найти всех сотрудников в иерархии подчинения Ивана Иванова (включая его самого). Для каждого вывести отдел, роль, проекты и задачи.

**Как работает запрос:**

Ядро запроса — **рекурсивное обобщённое табличное выражение** (`WITH RECURSIVE`):

1. **Базовый случай (anchor):** `SELECT ... FROM Employees WHERE EmployeeID = 1` — начальная точка рекурсии, сам Иван Иванов.

2. **Рекурсивный случай:** `SELECT e.* FROM Employees e JOIN subordinates s ON e.ManagerID = s.EmployeeID` — на каждой итерации находит сотрудников, чей `ManagerID` совпадает с `EmployeeID` из предыдущей итерации. Рекурсия продолжается, пока находятся новые строки.

   - Итерация 1: находит прямых подчинённых Иванова (Петров, Светлова, Андреев, Еленова) — 4 сотрудника
   - Итерация 2: находит подчинённых этих четверых (Алексеев, Олегов, Игорев, Мариева, Татеева, ...) — ещё сотрудники
   - Итерация 3: находит подчинённых третьего уровня (Кристинина, Викторов, Николаев, Сергеев, ...)
   - Итерация 4: новых подчинённых нет — рекурсия завершается

3. **Финальный SELECT** соединяет результат рекурсии с:
   - `Departments` (INNER JOIN) — название отдела
   - `Roles` (INNER JOIN) — название роли
   - `Projects` (LEFT JOIN по DepartmentID) — проекты отдела сотрудника
   - `Tasks` (LEFT JOIN по EmployeeID = AssignedTo) — задачи сотрудника

   `LEFT JOIN` для Projects и Tasks гарантирует, что сотрудники без проектов/задач не исчезнут из результата — вместо этого будет NULL.

4. **`GROUP_CONCAT(DISTINCT ... ORDER BY ... SEPARATOR ', ')`** — агрегирует несколько проектов/задач в одну строку. `DISTINCT` предотвращает дублирование (которое может возникнуть из-за множественных JOIN), `ORDER BY` обеспечивает детерминированный порядок.

5. **`GROUP BY`** по всем неагрегированным полям — необходим для корректной работы `GROUP_CONCAT`.

Результат: 30 строк (все сотрудники компании, так как все подчиняются Иванову), отсортированные по имени.

---

### Задача 2

**Условие:** То же, что задача 1, плюс: количество задач каждого сотрудника и количество его прямых подчинённых (без учёта подчинённых подчинённых).

**Как работает запрос:**

Запрос расширяет решение задачи 1 двумя дополнениями:

1. **CTE `direct_sub_count`** — предварительно подсчитывает прямых подчинённых для каждого менеджера:
   ```sql
   SELECT ManagerID, COUNT(*) AS TotalSubordinates
   FROM Employees WHERE ManagerID IS NOT NULL
   GROUP BY ManagerID
   ```
   Это отдельная агрегация по всей таблице Employees, не зависящая от рекурсии. Например, у Иванова (ID=1) — 4 прямых подчинённых, у Светловой (ID=3) — 7.

2. **`COUNT(DISTINCT t.TaskID)`** — подсчитывает уникальные задачи. `DISTINCT` важен, потому что LEFT JOIN с Projects может создать дубликаты строк задач.

3. **`LEFT JOIN direct_sub_count ds ON s.EmployeeID = ds.ManagerID`** — присоединяет подсчёт подчинённых. LEFT JOIN, потому что у большинства сотрудников подчинённых нет.

4. **`COALESCE(ds.TotalSubordinates, 0)`** — заменяет NULL на 0 для сотрудников без подчинённых. Без COALESCE в столбце были бы NULL-значения.

Ключевое отличие от задачи 1: `ds.TotalSubordinates` добавлен в `GROUP BY`, так как это неагрегированное поле из LEFT JOIN.

**Условие:** Найти сотрудников с ролью «Менеджер» (RoleID = 1), у которых есть подчинённые. Подсчитать всех подчинённых рекурсивно (включая подчинённых подчинённых).

**Как работает запрос:**

Этот запрос принципиально отличается от предыдущих — рекурсия запускается от каждого менеджера отдельно:

1. **Рекурсивный CTE `subordinates`** — содержит дополнительное поле `ManagerRoot`, которое сохраняет ID исходного менеджера на всех уровнях рекурсии:
   - **Anchor:** выбирает всех сотрудников с `RoleID = 1` (менеджеры). Каждый становится корнем своего дерева (`EmployeeID AS ManagerRoot`).
   - **Рекурсивный шаг:** находит подчинённых, сохраняя `s.ManagerRoot` — так каждый найденный сотрудник «помнит», от какого менеджера началась цепочка.
   - **`WHERE e.EmployeeID != s.ManagerRoot`** — предотвращает включение самого менеджера в список своих подчинённых (защита от зацикливания).

2. **CTE `total_sub`** — подсчитывает подчинённых для каждого `ManagerRoot`, исключая самого менеджера (`WHERE EmployeeID != ManagerRoot`).

3. **Финальный SELECT** — соединяет менеджеров с их подсчётами через `JOIN total_sub` (INNER JOIN, не LEFT — нужны только менеджеры с подчинёнными). Условие `WHERE ts.TotalSubordinates > 0` дополнительно гарантирует фильтрацию.
